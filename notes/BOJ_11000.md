# 11000. 강의실 배정

- **출처 / 링크:** [백준 11000](https://www.acmicpc.net/problem/11000)
- **난이도·태그:** Gold V — 그리디, 정렬, 우선순위 큐
- **상태:** 학습 정리

## 왜 어려웠는지 / 막힌 지점

- 문제 본질(겹치면 방이 더 필요하고, 안 겹치면 재사용)은 잡았는데, **“어떤 강의실을 고를지”**를 최적으로 줄이지 못했다.
- `ArrayList`로 강의실마다 끝나는 시각을 들고 있으면서 **앞에서부터 전부 훑는 방식**이면, 수업마다 O(강의실 수)라서 전체가 **O(N²)** 에 가깝게 불어난다.
- 실제로 필요한 건 “지금 비는 방” 중 **가장 빨리 끝난 방 하나**만 보면 되는데, 그걸 **최솟값 하나**로 압축하는 단계가 빠졌다.

## 잘 잡았던 것

- **시작 시간 기준 정렬**로 시간 순 처리 → 이후 선택만 올바르면 그리디가 성립하는 전제를 맞춤.
- 강의실 전체가 아니라 **각 방의 “현재 종료 시각”만** 들고 가겠다는 모델링.
- 재사용 조건 `이전 수업 종료 ≤ 다음 수업 시작`을 코드로 정확히 쓴 점(끝난 직후 다음 시작 허용).

## 시도했던 것들

- 강의실 목록을 리스트로 두고, 새 수업마다 리스트를 순회하며 넣을 방을 찾기 → 정답 논리는 맞아도 **자료구조가 병목**.

## 최종 풀이 요약

1. 수업을 **시작 시각 오름차순**으로 정렬한다. (시작이 같으면 종료로 tie-break해도 무방한 경우가 많다.)
2. **최소 힙(우선순위 큐)**에 “현재 사용 중인 각 강의실의 종료 시각”만 넣는다.
3. 현재 수업 `(s, t)`에 대해, 힙의 최댓값이 아니라 **최소 종료 시각**이 `s` 이하이면 그 방을 재사용(팝)하고, 아니면 새 방이 필요한 셈으로 그냥 `t`를 넣는다.
4. 처리 중 힙 크기의 최댓값이 필요한 강의실 수의 최솟값과 같다.

한 줄로는 **“가장 빨리 끝나는 강의실만 본다”**는 그리디 + **최솟값 유지**가 `PriorityQueue`로 해결된다.

## 새로 알게 된 것 / 다음에 쓸 것

- 그리디가 맞아도 **자료구조가 틀리면** 같은 아이디어로도 시간 초과가 난다.
- “전체를 다 볼 필요 없이 최소/최대 하나만 보면 된다”면 **힙·트리· multiset** 후보를 먼저 떠올릴 것.
