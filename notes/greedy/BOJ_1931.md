# 1931. 회의실 배정

- **출처 / 링크:** [백준 1931](https://www.acmicpc.net/problem/1931)
- **난이도·태그:** Gold V — 그리디, 정렬
- **상태:** 학습 정리

## 왜 어려웠는지 / 막힌 지점

- 회의를 **끝나는 시각 기준으로 고르는 그리디**는 잡았는데, **시작 시각과 끝 시각이 같은** 회의(길이 0 구간)를 comparator에서 빼먹으면 같은 줄에 있는 회의들의 상대 순서가 불안정해질 수 있다.
- `Comparator`에서 **첫 번째 키만** 두면(`finishAt`만) 동률일 때 어떤 회의가 먼저 나올지가 명확하지 않다. `thenComparing`으로 **두 번째 정렬 기준**을 둘 수 있다는 걸 몰랐다.

## 잘한 점

- **우선순위 큐**로 “끝나는 시각이 빠른 순”을 유지하겠다는 방향을 잡았다.
- IDE에서 `PriorityQueue` 생성자와 `Comparator` API를 직접 열어보며 `Comparator`를 지정해서 `PriorityQueue`를 선언했다.

## 시도했던 것들

- `finishAt`만 비교하는 `PriorityQueue` 선언 → 시작=끝인 회의까지 포함한 **완전한 순서**가 필요해 `thenComparingInt(Meeting::startAt)`를 추가.

## 최종 풀이 요약

1. 각 회의를 `(시작, 끝)`으로 받는다.
2. **끝 시각 오름차순**, 같으면 **시작 시각 오름차순**으로 정렬된 순서와 같게 `PriorityQueue`를 구성한다.
3. 큐에서 하나씩 꺼내며, 직전에 선택한 회의의 **끝 시각 ≤ 현재 회의의 시작 시각**이면 회의 하나 더 잡을 수 있다 → 카운트 증가, 끝 시각 갱신.

(배열로 받아 `Arrays.sort`만 해도 동일한 그리디다. PQ는 “항상 끝이 가장 이른 후보를 꺼낸다”는 구현 선택.)

## 새로 알게 된 것 / 다음에 쓸 것

- `Comparator.comparingInt(...).thenComparingInt(...)`로 **동점 처리**를 명시하면, 길이 0 회의·같은 끝 시각 등에서 순서가 흔들리지 않게 할 수 있다.
