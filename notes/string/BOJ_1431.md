# 1431. 시리얼 번호

- **출처 / 링크:** [백준 1431](https://www.acmicpc.net/problem/1431)
- **난이도·태그:** Silver III — 문자열, 정렬
- **상태:** 학습 정리

## 왜 어려웠는지 / 막힌 지점

- 정렬 기준은 머릿속으로는 세 단계(길이 → 자릿수 합 → 사전순)로 잡혔는데, **`Arrays.sort`의 `Comparator` 문법**에서 막혔다.
- 람다 안에서 `Integer.compare.reverseOrder`처럼 **존재하지 않는 호출**을 섞거나, `Comparator.comparing(a, b)`처럼 **쓰임이 다른 API**를 붙이는 식으로 헷갈렸다.

## 잘한 부분

- `getSum()`으로 문자열 안의 **숫자 자릿만 골라 합**을 구하는 로직은 문제 조건과 잘 맞게 짰다.
- 정렬 우선순위 자체(길이 짧은 것 → 합 작은 것 → 같으면 문자열 비교)는 올바른 방향이었다.

## 시도했던 것들

- 긴 람다 블록으로 `if` 분기마다 `return` 하려다 문법·API가 꼬임.
- 주석 처리된 버전처럼 `Integer.compare`, `getSum(a) - getSum(b)`, `a.compareTo(b)`로 직접 비교하는 방식도 가능하지만, 최종은 **`Comparator` 체이닝**으로 정리.

## 최종 풀이 요약

```
//        Arrays.sort(strs, (String a, String b) -> {
//            if(a.length() != b.length()) {
//                return Integer.compare(a.length(), b.length());
//            } else if(getSum(a) != getSum(b)) {
//                return getSum(a) - getSum(b);
//            } else {
//                return a.compareTo(b);
//            }
//        });

        Arrays.sort(strs,
                Comparator.comparingInt(String::length)
                        .thenComparingInt(s -> getSum(s))
                        .thenComparing(Comparator.naturalOrder())
        );
```        

1. `Comparator.comparingInt(String::length)` — **길이 오름차순**
2. `.thenComparingInt(s -> getSum(s))` — 길이가 같으면 **자릿수 합 오름차순**
3. `.thenComparing(Comparator.naturalOrder())` — 그다음 **사전순(기본 문자열 순서)**

한 줄로 조건을 이어 붙이면 분기 실수와 문법 오류가 줄어든다.

## 새로 알게 된 것 / 다음에 쓸 것

- 다중 기준 정렬은 **`comparing` / `thenComparing` 체인**이 읽기도 쉽고 실수도 적다.
- `getSum`처럼 보조 키는 `thenComparingInt(메서드 참조 또는 람다)`로 넘기면 된다.
