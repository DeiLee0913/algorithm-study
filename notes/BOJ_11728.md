# 11728. 배열 합치기

- **출처 / 링크:** [백준 11728](https://www.acmicpc.net/problem/11728)
- **난이도·태그:** Silver V — 정렬, 두 포인터
- **상태:** AC (메모리 348504 KB, 시간 1576 ms — 제출 기준)
- **풀이 위치:** `백준/Silver/11728. 배열 합치기/`

## 왜 어려웠는지 / 막힌 지점

초반에는 "합쳐서 정렬"만 생각해서 `List` 기반으로 접근했고, 연산 비용을 충분히 따지지 못했다.

- `remove()` 같은 중간 삭제 연산이 비싼데도 반복적으로 쓰는 실수를 했다.
- `stream().sorted()`는 편하지만, 새 리스트를 만들어 메모리를 더 쓰는 점을 간과했다.
- 문제 태그에 `두 포인터`가 있는데도, 처음엔 태그에 맞는 선형 병합 관점으로 접근하지 못했다.

## 시도했던 것들

- `List`에 전부 담아서 `stream().sorted()`로 정렬 후 출력
- 중간에 `remove()`를 쓰는 흐름으로 구현했다가 비용 문제를 인지하고 `get(i)` 중심으로 수정

## 최종 풀이 요약

현재 제출 코드는 다음 흐름이다.

1. 두 배열 원소를 하나의 `List<Integer>`에 넣는다.
2. `stream().sorted()`로 정렬한다.
3. `StringBuilder`로 출력 문자열을 모아 한 번에 출력한다.

정답은 맞지만, 입력이 큰 문제에서는 자료구조/연산 선택이 성능에 바로 영향을 준다는 점을 체감했다.

## 새로 알게 된 것 / 다음에 쓸 것

- `List.remove(index)`는 내부 이동이 발생해 반복 사용 시 비용이 크다.
- 코테에서 정렬이 필요하면 기본은 `Collections.sort(list)` 또는 `list.sort(null)`이 실수 적다.
- `stream().sorted()`는 가독성은 좋지만 새 컬렉션 생성 비용이 있어 대용량 입력에서는 주의한다.
- 정렬 방향 정리:
  - 오름차순: `Integer.compare(a, b)`
  - 내림차순: `Integer.compare(b, a)` 또는 `Comparator.reverseOrder()`
- 같은 문제를 다시 풀면 `두 포인터`로 O(N+M) 병합 풀이를 먼저 시도해 볼 것.

## 참고한 것

- Java `List` 연산 비용과 정렬 방식 차이를 다시 정리하면서 복습했다.
