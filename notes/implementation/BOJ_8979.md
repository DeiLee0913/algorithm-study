# 8979. 올림픽

- **출처 / 링크:** [백준 8979](https://www.acmicpc.net/problem/8979)
- **난이도·태그:** Silver V — 구현, 정렬
- **상태:** AC

## 왜 어려웠는지 / 막힌 지점

핵심은 단순 정렬이 아니라, **정렬 방향 + 동순위 처리 + 인덱스/순위 변환**을 동시에 맞춰야 한다는 점이었다.

- 문제는 금/은/동 메달이 **많은 나라가 앞**인데, 초기에 오름차순 비교를 넣어 정렬 자체가 반대로 됐다.
- 동순위를 `rank++`로 처리하면 안 되고, **이전 원소와 같은 메달 수면 rank 유지**가 필요했다.
- 배열 인덱스는 0-based인데 순위는 1-based라서 `rank = i`로 두면 한 칸씩 밀리는 실수가 났다.

## 시도했던 것들

- `Integer.compare(a.gold, b.gold)` 중심의 오름차순 비교 → 메달 적은 나라가 앞에 와서 오답
- 동순위인데도 순위를 계속 증가시키는 로직 → 공동 순위 조건 위반
- `i == 0`인 첫 원소에서 출력 조건을 누락하는 분기 구조 → 특정 케이스에서 출력 실패

## 최종 풀이 요약

1. 메달 기준으로 **내림차순 정렬**한다. (금 > 은 > 동 우선)
2. 순회하면서 이전 나라와 메달 3개가 모두 같으면 `rank`를 유지한다.
3. 다르면 `rank = i + 1`로 갱신한다. (`i`는 0-based)
4. 현재 나라 번호가 `K`이면 그 시점의 `rank`를 출력한다.

## (a, b) vs (b, a) 정렬이 왜 다른지

`Comparator`에서 `compare(x, y)`의 반환값 의미를 먼저 기억하면 헷갈림이 줄어든다.

- 음수: `x`를 `y`보다 앞에 둔다
- 양수: `x`를 `y`보다 뒤에 둔다
- 0: 순서 동일로 정렬 전의 상대 순서(원래 입력 순서)가 유지된다.

`Integer.compare(x, y)`는 다음을 반환한다.

- `x < y`면 음수
- `x == y`면 0
- `x > y`면 양수

그래서:

- `Integer.compare(a, b)`는 **작은 값이 앞**으로 가는 오름차순
- `Integer.compare(b, a)`는 비교 대상을 뒤집었기 때문에 **큰 값이 앞**으로 가는 내림차순

이 문제처럼 "큰 값 우선"이면 `compare(b, a)` 패턴을 기본으로 떠올리면 된다.

## 새로 알게 된 것 / 다음에 쓸 것

- 동률 규칙은 무엇이고 어떻게 다루어야 하는가
- `Comparator`는 함수 이름보다 **반환값(음수/양수)의 의미**로 읽어야 실수가 줄어든다
- 순위 문제는 단순 카운트가 아니라 **조건 기반 갱신**이다
