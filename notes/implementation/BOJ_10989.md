# 10989. 수 정렬하기 3

- **출처 / 링크:** [백준 10989](https://www.acmicpc.net/problem/10989)
- **난이도·태그:** Bronze I — 정렬
- **상태:** AC (메모리 337004 KB, 시간 1584 ms — 제출 기준)
- **풀이 위치:** `백준/sorting/10989. 수 정렬하기 3/`

## 왜 어려웠는지 / 막힌 지점

2750번처럼 **비교 정렬 + 라이브러리**

- `int[]`에 넣고 `Arrays.sort` → 메모리는 나아져도, **O(N log N)** 이고 입출력까지 합치면 **시간 제한에서 버거울 수 있다**(환경·구현에 따라 TLE).

즉 “정렬” 문제지만 **값의 범위가 1 ~ 10,000으로 좁다**는 조건을 먼저 읽어야 하는 유형이다.

## 시도했던 것들

- `Integer[]`에 담아 `Arrays.sort` → **메모리 초과**
- `int[]`에 담아 `Arrays.sort` → **시간 초과**

## 최종 풀이 요약

**카운팅 정렬(counting sort)** 을 쓴다.

- 크기 `10001`짜리 `int[] counts`에, 각 숫자 `i`가 등장한 횟수를 `counts[i]`에 누적한다.
- `i`를 1부터 10000까지 돌면서 `counts[i]`만큼 `i`를 출력 순서로 이어 붙인다 → **시간 O(N + K)** (여기서 K = 10000), 추가 메모리는 카운트 배열 정도.

출력은 한 줄씩 `System.out.println` 하면 호출이 너무 많아져서 **한 번에 모아 쓰는 `StringBuilder`** 에 붙인 뒤 마지막에 한 번 출력하는 쪽이 안전하다.

## 새로 알게 된 것 / 다음에 쓸 것

- **데이터 범위가 작을 때** “비교 정렬”만 고집하지 말고, **도수·카운팅** 가능 여부를 먼저 본다.
- **대량 출력**은 버퍼(`StringBuilder` 등)로 묶는 게 기본이다.
- 같은 “정렬” 태그라도 **N·값 범위·메모리 제한**에 따라 쓸 수 있는 알고리즘이 완전히 달라질 수 있다.
