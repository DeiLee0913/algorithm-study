# 9659. 트리 순회 (정올)

- **출처 / 링크:** [정올 9659 — 트리 순회](https://jungol.co.kr/problem/9659)
- **난이도·태그:** 실버 3 — 트리, DFS
- **상태:** AC
- **메인 아이디어(폴더):** graph
- **보조 태그:** dfs, tree, parent-array
- **풀이 위치:** `graph/JO_9659.java`

### 잘한 점

- (2)(3)은 **자손 방향만** 가고, (4)는 **부모 방향까지** 가야 한다는 차이를 먼저 나눠 두었다.
- 자손 탐색은 `parents[current] != next`, 전체 탐색은 `prev != next`로 역방향 차단 조건을 분리해 구조를 단순하게 잡았다.
- 입력은 `cid → pid` 한 방향이지만, (4)를 위해 `adjs`를 **양방향**으로 연결해 두었다.

### 왜 어려웠는지 / 막힌 지점

- 처음에는 네 가지 질의를 한꺼번에 생각하다 보니 탐색 방향·조건이 뒤섞여서 어렵게 느껴졌다.
- (2)(3) **자손**과 (4) **임의 노드**를 같은 DFS로 처리하려 하면, 부모로 되돌아가는 간선을 막는 기준이 달라져서 헷갈린다.
  - 전체: 시작점 기준으로 `prev != next`로 **직전에 온 간선만** 막으면 부모·형제 방향으로도 전개할 수 있다.
- (4)는 부모 쪽으로도 전개할 수 있으므로, 입력 때 `adjs[pid].add(cid)`뿐 아니라 `adjs[cid].add(pid)`도 넣어 **양방향 인접 리스트**가 필요하다. 한 방향만 저장하면 (4)에서 위쪽 노드를 탐색할 수 없다.
- 양방향으로 두면 (2)(3)에서 **이웃 중 누가 부모인지**가 바로 안 보여서 막혔다. AI 힌트로, 루트에서 한 번 DFS하며 `parents[]`를 채워 두고 `parents[current] != next`일 때만 아래로 내려가는 방식으로 정리했다.
- (4)는 BFS로도 풀 수 있지만, DFS에 `prev`만 넘기면 같은 최댓값을 구할 수 있어 최종 코드는 DFS로 통일했다.

### 최종 풀이 요약

1. **루트와의 거리:** 루트 `1`에서 DFS(`fromRoot`). `depths[v]`에 깊이를 저장하고, 동시에 `parents[v]`를 채운다. 출력은 `depths[target]`.
2. **본인 포함 자손 수:** `target`에서 `parents[current] != next`인 이웃만 DFS(`fromTarget`). 방문할 때마다 `num_of_offsprings++` (자기 자신 포함).
3. **가장 먼 자손과의 거리:** 같은 DFS에서 `distant`를 넘기며 `max_distant` 갱신.
4. **가장 먼 노드와의 거리:** `target`에서 `fromTargetToAnywhere(0, target, 0)`처럼 `prev`를 넘기며 모든 방향 DFS. `prev != next`일 때만 진행하고, `max_distant_to_anywhere`에 최댓값 저장.

### 새로 알게 된 것 / 다음에 쓸 것

- 트리 DFS에서 “자손만”과 “전체”는 **막는 간선의 기준**이 다르다. 부모 배열로 아래만 가기 vs `prev`로 되돌아가기만 막기.
- 루트 거리·부모 정보는 **한 번의 루트 DFS**로 전처리해 두면 (2)(3)이 깔끔해진다.
- (4)처럼 임의 방향 최장 거리는 양방향 `adjs` + `prev` 파라미터 패턴이 자주 쓰인다.
- 전역 변수(`max_distant`, `num_of_offsprings` 등) 대신 반환값·인자로 모으면 테스트·재사용은 더 쉽지만, PS 한 파일 풀이에서는 현재 형태도 무방하다.

### 참고한 것

- AI 힌트: 양방향 `adjs`에서 자손만 내려갈 때 `parents[]`로 부모 이웃을 걸러내는 방식.
